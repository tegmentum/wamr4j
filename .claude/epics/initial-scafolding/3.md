# Task 004: Native Build System

## Summary

Establish comprehensive native build infrastructure for wamr4j by creating a Rust workspace in the wamr4j-native module, integrating WAMR 2.4.1 compilation with CMake, and setting up cross-compilation targets for all supported platforms (Linux/Windows/macOS x86_64/ARM64). The system must produce platform-specific native libraries that are automatically packaged into JAR artifacts and consumed by both JNI and Panama implementations.

## Acceptance Criteria

### Core Infrastructure
- [ ] Rust workspace configured in `wamr4j-native/` with proper Cargo.toml hierarchy
- [ ] WAMR 2.4.1 source integration with automated CMake-based compilation
- [ ] Cross-compilation targets for 6 platform/architecture combinations
- [ ] Maven build lifecycle integration with native compilation before Java compilation
- [ ] Automated native library packaging into platform-specific JAR resources

### Platform Support Matrix
- [ ] Linux x86_64 (primary development platform)
- [ ] Linux ARM64 (server deployment target)
- [ ] Windows x86_64 (desktop development support)
- [ ] Windows ARM64 (modern Windows devices)
- [ ] macOS x86_64 (Intel-based development machines)
- [ ] macOS ARM64 (Apple Silicon development machines)

### Build System Integration
- [ ] Native compilation triggered by `mvn compile` in wamr4j-native module
- [ ] Platform detection and appropriate native library selection at runtime
- [ ] Incremental builds that skip native compilation when no changes detected
- [ ] Native library artifacts published to Maven repository with platform classifiers

### WAMR Integration Requirements
- [ ] WAMR 2.4.1 source code vendored or fetched during build
- [ ] CMake configuration for WebAssembly runtime compilation
- [ ] Rust FFI bindings to WAMR C API functions
- [ ] Both JNI exports and Panama FFI exports from single shared library

## Technical Approach

### Rust Workspace Architecture
```toml
[workspace]
members = [
    "wamr-sys",           # WAMR C API bindings
    "wamr4j-core",        # Core Rust implementation
    "wamr4j-jni",         # JNI exports
    "wamr4j-panama"       # Panama FFI exports
]
resolver = "2"
```

### WAMR Source Integration
- **Git Submodule**: Vendor WAMR 2.4.1 as git submodule for reproducible builds
- **CMake Build Script**: Custom build.rs scripts for WAMR compilation with platform-specific flags
- **Feature Configuration**: Enable required WAMR features (interp, fast-interp, libc-wasi, lib-pthread)
- **Optimization Settings**: Release builds with size optimization for runtime distribution

### Cross-Compilation Strategy
```rust
// Platform-specific compilation targets
[target.x86_64-unknown-linux-gnu]
linker = "x86_64-linux-gnu-gcc"

[target.aarch64-unknown-linux-gnu]  
linker = "aarch64-linux-gnu-gcc"

[target.x86_64-pc-windows-gnu]
linker = "x86_64-w64-mingw32-gcc"

[target.x86_64-apple-darwin]
linker = "clang"
```

### Maven Integration Points
1. **Pre-compile Phase**: Native library compilation and WAMR building
2. **Process-resources Phase**: Copy native artifacts to appropriate JAR locations
3. **Package Phase**: Create platform-specific JAR artifacts with native libraries
4. **Install Phase**: Deploy artifacts with platform classifiers to repository

## Implementation Details

### WAMR Build Configuration
```cmake
# Core WAMR features for Java integration
set(WAMR_BUILD_INTERP 1)
set(WAMR_BUILD_FAST_INTERP 1)
set(WAMR_BUILD_JIT 0)           # Disable JIT for consistency
set(WAMR_BUILD_LIBC_WASI 1)
set(WAMR_BUILD_LIB_PTHREAD 1)
set(WAMR_BUILD_MEMORY_PROFILING 0)  # Optional features
set(WAMR_BUILD_PERF_PROFILING 0)
```

### Rust API Surface Design
```rust
// Core WAMR functionality exposed to both JNI and Panama
pub struct WamrRuntime {
    engine: *mut wasm_engine_t,
    store: *mut wasm_store_t,
}

// JNI export functions
#[no_mangle]
pub extern "C" fn Java_ai_tegmentum_wamr4j_jni_WamrRuntimeImpl_createRuntime() -> jlong;

// Panama FFI export functions  
#[no_mangle]
pub extern "C" fn wamr_runtime_create() -> *mut WamrRuntime;
```

### Platform Detection Logic
- Runtime platform detection using `System.getProperty("os.name")` and `System.getProperty("os.arch")`
- Native library loading with platform-specific naming conventions
- Fallback mechanisms for unsupported platforms
- Clear error messages when native libraries unavailable

### Build Orchestration Scripts
```bash
#!/bin/bash
# Cross-compilation script for CI/CD integration
platforms=("x86_64-unknown-linux-gnu" "aarch64-unknown-linux-gnu" 
           "x86_64-pc-windows-gnu" "x86_64-apple-darwin" "aarch64-apple-darwin")

for platform in "${platforms[@]}"; do
    cargo build --target "$platform" --release
    # Package into platform-specific JAR
done
```

## Risk Assessment

### Technical Risks
- **Cross-Compilation Complexity**: Toolchain setup varies significantly between platforms
  - *Mitigation*: Docker-based build environment, comprehensive documentation, fallback to manual builds
- **WAMR Version Compatibility**: API changes between WAMR versions breaking Rust bindings
  - *Mitigation*: Pin to specific WAMR 2.4.1 tag, automated API compatibility testing
- **Native Library Size**: WAMR compilation producing large binaries affecting JAR size
  - *Mitigation*: Optimization flags, optional feature disabling, separate platform artifacts

### Integration Risks
- **Maven Build Performance**: Native compilation significantly increasing build times
  - *Mitigation*: Incremental builds, parallel compilation, optional native skipping for development
- **Platform-Specific Bugs**: Native code behaving differently across platforms
  - *Mitigation*: Comprehensive automated testing on all platforms, defensive programming patterns
- **Native Library Loading**: Runtime failures when loading platform-specific libraries
  - *Mitigation*: Robust platform detection, clear error messages, graceful fallbacks

### Security Risks
- **Native Code Vulnerabilities**: Memory safety issues in Rust/WAMR integration
  - *Mitigation*: AddressSanitizer builds, fuzzing test integration, conservative API design
- **Supply Chain Security**: WAMR source integrity and reproducible builds
  - *Mitigation*: Git submodule with signed tags, checksum verification, offline build capability
